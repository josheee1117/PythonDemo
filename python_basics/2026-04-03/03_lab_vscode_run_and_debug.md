# 实验 2：运行按钮与断点调试

目标：不用讲一堆调试理论，先亲手完成一次最小调试闭环。

## 步骤 1：先直接运行，故意看到错误

打开文件：

```text
python_basics/2026-04-03/05_vscode_debug_demo.py
```

先不要改代码，直接运行一次：

```powershell
.\.venv\Scripts\python.exe python_basics\2026-04-03\05_vscode_debug_demo.py
```

你应该会看到断言失败。

## 步骤 2：打断点

在下面这一行左侧点一下，打一个断点：

```python
assert completion_rate == 1.0
```

## 步骤 3：从 VS Code 启动调试

有两种方式，任选一种：

- 打开 `Run and Debug` 面板，选择 `Day1 Debug Demo`
- 或者直接按 `F5`

项目里已经放好了 `.vscode/launch.json`，可以直接用。

## 步骤 4：停住后看变量

停在断点后，重点看这 3 个变量：

- `completed_steps`
- `total_steps`
- `completion_rate`

先不要猜，先看值。

## 步骤 5：修复问题并再次运行

修完以后，重新运行这个文件，直到断言通过。

今天不追求会所有调试技巧，只要你完成下面这条链路就够了：

```text
运行失败 -> 打断点 -> 看变量 -> 改代码 -> 再运行成功
```

## Java 对照

- 这和你在 IDEA 里打断点、看局部变量、继续执行，本质是同一套调试思路
- 不同点是 Python 变量类型更动态，所以“先看运行时值”会更重要

## 常见错误

- 还没运行失败过，就先凭感觉改代码
- 断点打对了，但启动的是普通运行不是调试
- 只看报错信息，不看断点停住时的变量值

## 完成标准

- 你真的看到了第一次失败
- 你真的打了断点并停住
- 你能说出这次错误是哪个变量不对
- `05_vscode_debug_demo.py` 最终运行通过
